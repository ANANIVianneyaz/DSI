4. Architecture Multi-Sites

Approches possibles

Sous-domaines

Chaque application a son propre sous-domaine (ex: app1.ministere.bj, app2.ministere.bj)

app1.ministere.bj → 123.45.67.89:5000
app2.ministere.bj → 123.45.67.89:5001

Sous-répertoires

Toutes les applications sont accessibles via le même domaine mais différents chemins (ex: ministere.bj/app1, ministere.bj/app2)

Exemple de configuration Nginx

server {
    server_name app1.ministere.bj;

    location / {
        proxy_pass http://localhost:5000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

server {
    server_name app2.ministere.bj;

    location / {
        proxy_pass http://localhost:5001;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

Cette configuration permet de router les requêtes vers différentes applications en fonction du sous-domaine.

Avantages de cette architecture

Économique

Un seul serveur pour plusieurs applications

Centralisé

Gestion unifiée des ressources

Évolutif

Ajout facile de nouvelles applications
